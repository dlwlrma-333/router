# 词元计算实施设计

本文档是 [词元计算方案](./词元计算方案.md) 的实现落地版，目标是把“本地估算优先、供应商独立估算”的新策略接入当前 Router 代码库，并尽量减少对现有结算逻辑的扰动。

## 1. 实施目标

第一阶段只解决一件事：

- **把文本请求的本地 token 估算接入预扣主链路**

明确不在第一阶段处理：

- 最终 usage 结算逻辑重构
- 对外 `/v1/messages/count_tokens` 接口
- 全量多模态精细估算
- 任务链路 / 视频 / 音频 / 图片链路统一接入
- 改写后重估

## 2. 当前接线点

当前文本预扣主链路在：

- [text.go](/Users/liuxin2/Workspace/opensource/router/internal/relay/controller/text.go:52)

关键调用顺序：

```text
RelayTextHelper
  -> getAndValidateTextRequest
  -> resolveChannelTextUpstream
  -> ResolveChannelModelPricing
  -> tokenestimate.Estimate
  -> ComputeTextPreConsumedQuota
  -> reserveRelayQuota
  -> preConsumeQuota
  -> convertTextRequestForUpstream
  -> getRequestBody
  -> adaptor.DoRequest
  -> adaptor.DoResponse
  -> postConsumeQuota
```

旧实现里 `promptTokens` 曾来自 `getPromptTokens`。当前已经切到 `tokenestimate.Estimate(...)`，但新的估算实现仍有进一步收敛空间。

当前主要问题是：

1. 逻辑按 `relayMode` 分支，而不是按供应商家族和输入结构分层
2. 当前大量复用 OpenAI tokenizer 逻辑
3. 没有统一表达“估算来源”和“精度等级”
4. 不利于后续扩展工具、媒体、协议补偿和改写后重估

## 3. 第一阶段改造原则

### 3.1 保持结算逻辑不变

第一阶段只替换：

- **预扣前 `promptTokens` 的估算来源**

不修改：

- `billing.ComputeTextQuota`
- `postConsumeQuota`
- usage 解析逻辑
- adaptor 侧响应回填逻辑

### 3.2 词元估算不依赖统一 bridge 输入层

当前代码里，文本请求在进入 `RelayTextHelper` 后，部分路径已经被统一解析为：

- [GeneralOpenAIRequest](/Users/liuxin2/Workspace/opensource/router/internal/relay/model/general.go:24)

其中：

- OpenAI `/v1/chat/completions`
- OpenAI `/v1/responses`
- Anthropic `/v1/messages`（当前通过桥接转换）

但词元估算不应建立在这层桥接之上。

原因：

1. 一旦 bridge 漏字段，估算输入就已经失真
2. Anthropic / OpenAI / Gemini 会通过共享 bridge 逻辑产生相互影响
3. 词元估算需要“各供应商独立演进”，而不是“共享转换后结构”

因此，第一阶段目标调整为：

- **各供应商估算器读取各自原生请求**
- **上层只统一消费 `EstimateResult`**

当前代码里保留 bridge 仅用于请求转发兼容，不作为词元估算的基础抽象。

### 3.3 第一阶段只覆盖文本主路径

第一阶段支持：

1. `chat/completions`
2. `responses`
3. `messages`

对应模式：

- `relaymode.ChatCompletions`
- `relaymode.Responses`
- `relaymode.Messages`

第一阶段先不接：

- `Completions`
- `Moderations`
- 图像 / 音频 / 视频控制器

## 4. 目标模块结构

建议新增独立模块：

```text
internal/tokenestimate
```

第一阶段文件建议如下：

```text
internal/tokenestimate/
  types.go
  estimate.go
  openai.go
  anthropic.go
  gemini.go
  heuristic.go
```

职责：

### 4.1 `types.go`

定义统一输入输出结构。

建议结构：

```go
package tokenestimate

type Precision string

const (
    PrecisionExact     Precision = "exact"
    PrecisionHeuristic Precision = "heuristic"
    PrecisionCoarse    Precision = "coarse"
)

type EstimateRequest struct {
    RelayMode       int
    ChannelProtocol int
    UpstreamMode    int
    UpstreamPath    string
    Model           string
    RawBody        []byte
}

type EstimateResult struct {
    PromptTokens int
    Source       string
    Precision    Precision
    Estimator    string
}
```

说明：

- `EstimateRequest.RawBody` 是估算入口的主输入
- 估算器应自己解析所需的原生请求体
- 如后续需要，也可以增加 `Parsed any` 作为可选缓存，但不能强制所有供应商共享同一 DTO

### 4.2 `openai.go`

负责 OpenAI / Responses 原生请求的抽取和估算。

第一阶段要求至少提取：

- messages role
- text content
- name
- tool name
- tool description
- tool parameters
- `max_tokens`

说明：

- 优先使用现有 OpenAI tokenizer
- 不要求第一阶段引入完整媒体抽象
- 可以先忽略图片/文件等复杂输入，或给后续预留字段

### 4.3 `anthropic.go`

负责：

- Anthropic `messages` 原生请求解析
- `system`
- `messages[].content[]`
- `tool_use`
- `tool_result`
- `tools`
- Anthropic 文本 heuristic 估算

说明：

- 不先桥接成 `GeneralOpenAIRequest`
- 估算器直接面对 Anthropic 原生 block 结构
- 这样 bridge 出问题不会污染估算正确性

### 4.4 `gemini.go`

负责：

- Gemini 原生请求解析
- 文本 / parts / tools 抽取
- Gemini heuristic 估算

### 4.5 `heuristic.go`

负责：

- Anthropic heuristic
- Gemini heuristic
- unknown heuristic

第一阶段不需要把 `new-api` 的实现逐字复制过来，但建议沿用它的设计思路：

- 英文单词
- 数字串
- CJK
- 标点
- emoji
- 空格 / 换行

并按 provider family 使用不同权重。

### 4.6 `estimate.go`

作为统一入口：

```go
func Estimate(req EstimateRequest) (EstimateResult, error)
```

内部流程建议：

1. 结合 `channel protocol / upstream mode / model / path` 选择供应商估算器
2. 由对应估算器直接解析原生请求
3. 返回统一结果

## 5. 第一阶段估算器选择规则

第一阶段不做复杂 registry，先用简单且明确的分发函数。

建议函数：

```go
func selectTextEstimator(model string) estimatorKind
```

规则如下：

### 5.1 OpenAI family

匹配：

- `gpt-`
- `o1`
- `o3`
- `o4`

结果：

- `source=local_openai_tokenizer`
- `precision=exact`
- `estimator=openai_exact`

### 5.2 Anthropic family

匹配：

- `claude`

结果：

- `source=local_anthropic_heuristic`
- `precision=heuristic`
- `estimator=anthropic_heuristic`

### 5.3 Gemini family

匹配：

- `gemini`

结果：

- `source=local_gemini_heuristic`
- `precision=heuristic`
- `estimator=gemini_heuristic`

### 5.4 Unknown family

其他模型：

- `source=local_unknown_heuristic`
- `precision=heuristic`
- `estimator=unknown_heuristic`

说明：

- 第一阶段可以先按 `channel protocol + relay mode + model family` 做简单分类
- 第二阶段再结合 `provider / upstream endpoint / channel setting` 收紧策略

## 6. 请求抽取设计

第一阶段按供应商独立提取。

### 6.1 OpenAI Chat / Responses

从 OpenAI 原生 `messages` / `input` 提取：

- `role`
- `name`
- string content
- array content 里的 text

同时提取 tools：

- `tool.function.name`
- `tool.function.description`
- `tool.function.parameters`

### 6.2 Anthropic Messages

从 Anthropic 原生请求提取：

- `system`
- `messages[].role`
- `messages[].content[].text`
- `messages[].content[].tool_use.name`
- `messages[].content[].tool_use.input`
- `messages[].content[].tool_result.content`
- `tools`
- `max_tokens`

说明：

- 不依赖 Anthropic bridge
- 不要求第一阶段引入图片精确估算
- 先把文本和工具相关上下文纳入预扣估算

### 6.3 Gemini

第一阶段可先预留接口，后续逐步补全。

## 7. 当前实现与目标状态的差距

当前已经接入的实现与目标方案相比，主要差距有：

1. [`internal/relay/controller/helper.go`](/Users/liuxin2/Workspace/opensource/router/internal/relay/controller/helper.go) 在 `relaymode.Messages` 下仍先走 `anthropic.ParseMessagesRequestToGeneralOpenAIRequest(...)`

下一轮代码改造应优先清理这条依赖链。

## 8. 与现有代码的集成点

### 8.1 第一阶段替换点

当前接线点已经在：

- [text.go](/Users/liuxin2/Workspace/opensource/router/internal/relay/controller/text.go:111)

下一轮建议把入口参数改成：

```go
func estimatePromptTokens(rawBody []byte, textRequest *relaymodel.GeneralOpenAIRequest, meta *meta.Meta) (tokenestimate.EstimateResult, error)
```

在 [text.go](/Users/liuxin2/Workspace/opensource/router/internal/relay/controller/text.go:110) 调用。

目标改造：

```go
estimateResult, err := estimatePromptTokens(requestBody, textRequest, meta)
if err != nil { ... }
promptTokens := estimateResult.PromptTokens
meta.PromptTokens = promptTokens
```

说明：

- OpenAI / Responses 可直接使用 `textRequest`
- Anthropic 估算器应优先使用 `rawBody`
- `textRequest` 在这里退化为“现有主链路兼容参数”，不是唯一估算输入

### 8.2 日志字段

第一阶段建议在 `RelayTextHelper` 中增加 debug 日志：

- `prompt_token_estimator`
- `prompt_token_source`
- `prompt_token_precision`
- `estimated_prompt_tokens`

建议日志示例：

```text
[prompt_token_estimate] estimator=openai_exact source=local_openai_tokenizer precision=exact tokens=1234 model=gpt-5.4 path=/v1/responses
```

第一阶段先打日志，不必立刻把这些字段写入数据库。

## 9. 第一阶段测试计划

建议新增：

```text
internal/tokenestimate/estimate_test.go
internal/tokenestimate/extract_test.go
```

至少覆盖以下场景：

### 8.1 OpenAI chat 纯文本

断言：

- 走 `openai_exact`
- `precision=exact`
- token > 0

### 8.2 OpenAI responses 纯文本

断言：

- 走 `openai_exact`
- 能从 `input` / `prompt` / `tools` 抽到文本

### 8.3 Anthropic messages 纯文本

断言：

- 走 `anthropic_heuristic`
- `precision=heuristic`

### 8.4 OpenAI chat + tools

断言：

- tools 定义会影响估算结果

### 9.5 Anthropic messages + tools

断言：

- 不经过 messages bridge，tool 相关内容仍能进入估算
- `tool_use` / `tool_result` / `system` 都会影响估算结果

### 9.6 Unknown model fallback

断言：

- 走 `unknown_heuristic`

此外建议补一个控制器层集成测试：

- 比较当前控制器预扣路径与 OpenAI tokenizer 直接计数在典型 OpenAI chat 场景下的基本一致性

## 11. 建议的落地顺序

按最小风险排序：

1. 保留现有预扣接线，补正文档和目标约束
2. 为 `Estimate()` 增加原生 `RawBody` 输入
3. 拆出 Anthropic 原生估算器，摆脱 bridge 依赖
4. 保持 OpenAI / Responses 继续走现有 tokenizer 路径
5. 增加 Anthropic 原生请求单测
6. 清理 bridge 相关估算测试
7. 增加 debug 日志
8. 全量回归测试

## 12. 第一阶段完成标准

满足以下条件即可认为第一阶段完成：

1. 文本主链路预扣已走 `tokenestimate.Estimate(...)`
2. `/v1/chat/completions`、`/v1/responses`、`/v1/messages` 三条文本路径都能走新估算器
3. OpenAI 文本走 tokenizer，Anthropic / Gemini / unknown 走 heuristic
4. Anthropic 词元估算不依赖 messages bridge
5. 估算来源与精度等级可在日志中观察
6. 最终 usage 结算逻辑保持不变
