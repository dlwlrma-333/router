# 词元计算方案

本文档描述 Router 后续采用的“词元（token）计算”新方案，重点关注：

- 请求前预扣 / 预占所依赖的本地估算
- 最终真实 usage 结算与本地估算的边界
- 不同供应商、不同协议、不同输入类型的估算分层
- 估算来源、精度等级与回退行为的可观察性

边界：

- `YYC` 记账、法币折算、预占/结算账本：见 [扣费逻辑](./扣费逻辑.md)
- 模型价格、价格单位、复杂价格组件：见 [模型计费](./模型计费.md)

## 1. 核心结论

Router 的词元计算主策略调整为：

1. **本地估算优先**
   - 请求前预扣 / 预占不依赖远程 `count_tokens` 接口。
   - 主路径采用本地估算器完成输入 token 估算。

2. **最终结算仍以上游 usage 为准**
   - 请求完成后，优先使用上游真实 usage 做最终结算。
   - 本地估算主要服务于“请求进入前的额度控制”和“请求开始时的预扣”。

3. **远程 `count_tokens` 不再作为主路径依赖**
   - 某些上游即使支持官方 `count_tokens`，也不作为 Router 预扣主链路的必需条件。
   - 是否支持远程 count endpoint，只影响未来可选能力，不影响核心可用性。

4. **估算精度分层，而不是假设所有供应商都能精确计数**
   - OpenAI 文本类模型可做到相对精确。
   - Anthropic / Gemini / 其他供应商文本类模型先采用启发式估算。
   - 多模态输入大量场景只能给出粗估，不应伪装成“上游权威 token 数”。

## 2. 为什么切换到本地估算优先

旧思路里，Anthropic `/v1/messages/count_tokens` 被视为重要主路径。但在真实渠道环境里，这条路线有明显问题：

1. **渠道能力不稳定**
   - 官方 Anthropic 支持 `/v1/messages/count_tokens`
   - 第三方 Anthropic 兼容供应商不一定支持
   - 即使同为 `/v1/messages`，不同渠道能力也不同

2. **会把 Router 的核心预扣能力绑定到远程接口可用性**
   - 上游没开放 count endpoint
   - 上游接口限流
   - 上游偶发错误
   - count endpoint 与真正推理 endpoint 可用性不一致

   以上情况都不应导致 Router 无法执行预扣。

3. **预估和实际请求可能不走同一渠道**
   - 某个支持 count endpoint 的渠道可以帮忙估算
   - 但实际推理请求可能路由到另一个不支持该接口的渠道
   - 这种结果只能叫“参考估算”，不能叫“实际目标渠道的官方 token 数”

4. **本地估算覆盖面更广**
   - 不仅覆盖 Anthropic
   - 也能统一覆盖 OpenAI、Gemini、OpenAI-compatible 供应商
   - 更适合 Router 这种聚合网关场景

因此，新方案明确：

- **预扣主路径必须脱离远程 count endpoint**
- **远程 count endpoint 只能作为未来可选能力，不作为基础依赖**

## 3. 与最终结算的边界

Router 的词元计算必须始终区分两个阶段：

### 3.1 请求前预扣 / 预占

目标：

- 判断请求是否允许进入上游
- 估算本次请求需要预留多少额度
- 在并发条件下避免“明显透支后再失败”

特点：

- 可以接受“有界误差”
- 不要求与最终 usage 绝对一致
- 必须稳定、快速、独立于远程 count endpoint

### 3.2 请求后真实结算

目标：

- 按最终上游 usage 做真实扣费
- 对“预估值 - 实际值”做差额回补或补扣

特点：

- 优先信任上游返回 usage
- 本地估算只作为缺失 usage 时的兜底，不作为默认替代

因此，词元计算方案的目标不是：

- 用本地估算完全替代真实 usage

而是：

- 保证预扣主路径稳定可用
- 保证最终结算口径清晰
- 让估算来源、精度和回退行为可观察

## 4. 新方案的总体结构

### 4.1 分层结构

词元计算拆成四层：

1. **供应商原生抽取层**
   - 按供应商 / 协议原生结构分别提取：
     - 文本
     - 工具定义
     - 工具结果
     - 媒体输入
     - `max_tokens` / `max_output_tokens`
   - 不把词元估算建立在“先桥接成统一请求结构”之上

2. **文本计数层**
   - 对纯文本部分做 token 估算
   - 按供应商家族选择：
     - OpenAI tokenizer
     - Anthropic heuristic
     - Gemini heuristic
     - unknown fallback heuristic

3. **媒体估算层**
   - 对图片、音频、视频、文件等输入给出本地估算
   - 不同输入类型允许不同精度等级

4. **结果统一层**
   - 对上游不同估算器的结果统一输出
   - 对 chat/messages/responses 的封装开销、工具 schema / tool result / request wrapper 做各自估算器内的补偿

### 4.2 主调用链

文本链路目标形态：

```text
RelayTextHelper
  -> estimate request tokens from native provider request
  -> local token estimate
  -> ComputeTextPreConsumedQuota
  -> reserve / preConsumeQuota
  -> do request
  -> parse usage
  -> postConsumeQuota
```

含义：

1. 请求进入上游前，先通过本地估算器得到 `estimated_prompt_tokens`
2. 依据估算结果完成预扣 / 预占
3. 请求返回后，优先按上游真实 usage 重算并结算

## 5. 可借鉴的本地估算策略

新方案明确借鉴 `new-api` 的总体思路，但不直接照搬其全部实现细节。

核心借鉴点是：

- 本地估算优先
- 不依赖远程 `count_tokens`
- 最终统一的是估算结果，而不是请求对象本身

明确不照搬的部分是：

- 不把不同供应商请求先桥接成统一请求 DTO 再估算
- 不让 Anthropic / Gemini / OpenAI 在估算输入层互相共享桥接结果

### 5.1 文本部分

#### OpenAI 文本模型

策略：

- 使用真实 tokenizer
- 按模型名选择 codec
- 若模型 codec 未命中，则回退到默认 OpenAI tokenizer 基座

适用范围：

- `gpt-*`
- `o1*`
- `o3*`
- `o4*`
- 其他明确属于 OpenAI tokenizer 家族的文本模型

精度：

- `exact`

说明：

- 这里的“exact”是指“相对本地估算路径而言的高精度”
- 仍需叠加协议封装开销和工具开销补偿

#### Anthropic / Gemini / 其他非 OpenAI 文本模型

策略：

- 使用启发式估算器
- 按字符类型加权：
  - 英文单词
  - 数字串
  - CJK
  - 标点
  - 数学符号
  - URL 分隔符
  - emoji
  - 空格 / 换行

适用范围：

- `claude*`
- `gemini*`
- 其他不适合使用 OpenAI tokenizer 的文本模型

精度：

- `heuristic`

说明：

- 这是“可用于预扣”的近似值
- 不应宣称为上游官方精确 token 数

### 5.2 工具与结构化输入

本地估算不只统计用户文本，还应尽量把真正进入模型上下文的结构化内容纳入估算：

- message role
- name
- tool name
- tool description
- tool schema / parameters
- tool call input
- tool result content

这是预扣可用性的关键部分，不能只统计裸 prompt 文本。

但这些结构化内容应由各供应商原生抽取器分别处理，例如：

- Anthropic 直接读取 `system`、`messages[].content[]`、`tool_use`、`tool_result`
- OpenAI 直接读取 `messages`、`responses.input`、`tool_calls`
- Gemini 直接读取其原生 content parts

而不是先压平成同一套请求结构后再估算。

### 5.3 媒体输入

#### OpenAI 图片输入

策略：

- 参考 OpenAI 视觉模型公开规则
- 按模型家族、detail、patch/tile 规则估算

精度：

- `heuristic` 到 `exact` 之间，视模型而定

说明：

- 这是本地估算里相对最值得做精细化的一类多模态输入

#### 其他图片 / 音频 / 视频 / 文件

策略：

- 先允许使用保守固定值或经验值
- 后续逐步按供应商能力细化

精度：

- `coarse`

说明：

- 只能用于预扣与准入控制
- 不适合对外宣称为真实 token 数

## 6. 供应商 / 协议 / 模型分层

### 6.1 不按“是否支持远程 count endpoint”分主线

新方案不再把：

- `supports_prompt_token_count_api`

作为本地预扣主路径的关键分支条件。

原因：

- 预扣主路径必须独立于远程 count endpoint

### 6.2 按供应商家族与最终上游协议选估算器

估算器选择至少结合：

- `channel_protocol`
- `provider`
- `upstream endpoint`
- `model`
- `input type`

不能只按模型名前缀决定，也不能只按 relay mode 决定。

### 6.3 第一阶段支持范围

第一阶段建议重点支持：

1. OpenAI chat/completions 文本
2. OpenAI responses 文本
3. Anthropic messages 文本
4. Gemini 文本

第一阶段先不追求精细覆盖：

- 文件理解
- 视频理解
- 复杂音频输入
- 全量图片 URL/base64 精确对齐
- tool surcharge 的独立定价推导

## 7. 精度等级与来源标记

本地估算结果必须显式带上“来源”和“精度”，禁止隐式假装精确。

建议统一结果结构至少包含：

```go
type EstimateResult struct {
    PromptTokens     int
    CompletionTokens int
    TotalTokens      int
    Source           string
    Precision        string
    Estimator        string
}
```

建议约定：

- `precision=exact`
  - OpenAI tokenizer 主路径

- `precision=heuristic`
  - Anthropic / Gemini / unknown 文本启发式估算
  - OpenAI 图片规则估算

- `precision=coarse`
  - 文件 / 视频 / 通用多模态占位估算

来源示例：

- `local_openai_tokenizer`
- `local_anthropic_heuristic`
- `local_gemini_heuristic`
- `local_unknown_heuristic`
- `local_openai_image_rules`
- `local_media_coarse`

## 8. 对远程 count endpoint 的新定位

### 8.1 不作为预扣主路径依赖

即使某个 Anthropic 渠道支持：

- `/v1/messages/count_tokens`

也不作为请求前预扣的主依赖。

### 8.2 可作为未来的可选增强能力

未来如果要支持对外：

- `/v1/messages/count_tokens`

可以作为单独能力处理，但语义必须清楚：

1. 若指定了具体渠道，且该渠道支持官方 count endpoint：
   - 可以透传或代理该能力

2. 若未指定渠道，或目标渠道不支持：
   - 应明确返回“本地估算结果”或“不支持”
   - 不能伪装成官方 count 结果

### 8.3 远程 count endpoint 仅用于补充，不用于替代本地估算基础设施

远程 count endpoint 可以在未来用于：

- 管理后台渠道测试
- 渠道能力验证
- 指定渠道场景下的对外能力增强

但不应反过来主导 Router 的基础预扣架构。

## 9. 估算时机

### 9.1 第一估算点：下游请求解析后

第一阶段先在请求进入后、正式发往上游前做本地估算。

要求：

- 基于统一 `EstimateResult`
- 各供应商独立完成文本、工具和基础媒体抽取

### 9.2 长期目标：支持改写后重估

Router 当前可能在真正上游请求发送前做改写，例如：

- 模型映射
- 协议转换
- endpoint policy
- `image_url_to_base64`

因此长期目标应支持：

- **改写后重估**

即：

1. 初次估算用于快速预扣
2. 若改写导致输入显著变化，允许补充重估
3. 若重估值高于当前预扣值，允许补预扣

这也是 Router 与 `new-api` 场景的重要差异。

## 10. 推荐模块结构

建议把词元估算抽成独立模块，例如：

```text
internal/tokenestimate
```

推荐拆分：

```text
internal/tokenestimate/
  types.go
  estimate.go
  extract_openai.go
  extract_anthropic.go
  extract_gemini.go
  textcounter_openai.go
  textcounter_heuristic.go
  media.go
```

统一接口建议如下：

```go
type EstimateRequest struct {
    ChannelProtocol     int
    Provider            string
    DownstreamPath      string
    UpstreamPath        string
    Model               string
    Body                []byte
}

type EstimateResult struct {
    PromptTokens     int
    CompletionTokens int
    TotalTokens      int
    Source           string
    Precision        string
    Estimator        string
}

func Estimate(req EstimateRequest) (EstimateResult, error)
```

注意：

- 这是“本地估算接口”，不是“远程 count api 抽象”
- 不需要把远程 count endpoint 设计成主接口的一部分
- 各供应商估算器应直接消费原生请求 body 或原生 DTO，而不是消费桥接后的统一请求 DTO

## 11. 分阶段实施

### 11.1 第一阶段：落本地估算基础设施

目标：

- 建立独立的本地估算模块
- 替换预扣主路径对远程 count endpoint 的依赖假设

范围：

1. 新增统一 `EstimateResult`
2. 支持三类下游文本请求：
   - `/v1/chat/completions`
   - `/v1/responses`
   - `/v1/messages`
3. OpenAI 文本走 tokenizer
4. Anthropic / Gemini / unknown 文本走 heuristic
5. 输出统一日志字段：
   - `prompt_token_estimator`
   - `prompt_token_source`
   - `prompt_token_precision`

### 11.1.1 第一阶段的实现约束

第一阶段虽然可以先保留现有代码中的桥接逻辑用于请求转发，但词元估算本身应遵循以下约束：

1. OpenAI 估算器只读取 OpenAI / Responses 原生请求
2. Anthropic 估算器只读取 Anthropic `messages` 原生请求
3. Gemini 估算器只读取 Gemini 原生请求
4. 上层只统一消费 `EstimateResult`，不统一消费桥接后的请求对象

### 11.2 第二阶段：接入预扣主链路

目标：

- 本地估算正式驱动预扣 / 预占

范围：

1. 在预扣计算前调用本地估算
2. 将估算结果接入 `ComputeTextPreConsumedQuota`
3. 最终结算逻辑仍维持：
   - 优先上游 usage
   - 无 usage 时按现有兜底策略处理

### 11.3 第三阶段：支持改写后重估

目标：

- 让预扣更贴近最终上游请求

范围：

1. 在关键请求改写点后增加重估入口
2. 当估算值显著上升时允许补预扣
3. 保留“初次估算”和“重估”日志链路

### 11.4 第四阶段：可选远程 count 增强

目标：

- 在不影响本地主路径的前提下，补充指定渠道的官方 count 能力

范围：

1. 管理后台测试指定渠道 count endpoint
2. 若未来对外开放 `/v1/messages/count_tokens`，其返回必须标记：
   - `official`
   - `local_estimate`
   - `unsupported`

## 12. 方案约束

1. **禁止把远程 count endpoint 当成预扣主路径**
   - 这是本方案的首要约束

2. **禁止静默伪装精度**
   - 启发式估算必须明确标记为 `heuristic`
   - 粗估必须明确标记为 `coarse`

3. **禁止把本地估算直接等同于最终 usage**
   - 预扣与最终结算必须分层

4. **禁止只按模型名前缀做全部决策**
   - 模型名前缀只作为一个信号
   - 仍需结合供应商、协议、端点和输入类型

5. **多模态场景先接受分层精度**
   - 第一阶段不追求全量精确
   - 先把主链路稳定性做出来

6. **禁止把跨供应商 bridge 当成词元估算的基础输入层**
   - bridge 可用于转发兼容
   - 但词元估算必须以供应商原生请求为准

## 13. 预期收益

1. 预扣主路径不再依赖远程 `count_tokens` 能力，稳定性显著提升。
2. OpenAI、Anthropic、Gemini、OpenAI-compatible 等供应商可统一进入本地估算框架。
3. 估算来源、精度和回退行为可明确观察，不再混成黑盒。
4. 最终真实结算口径保持清晰：预扣靠本地估算，结算优先看真实 usage。
5. 为后续图片、文件、音频、视频、多协议转换和改写后重估提供清晰扩展位。
