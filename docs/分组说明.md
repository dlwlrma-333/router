# 分组说明

本文档说明 Router 中“分组（group）”的作用、数据结构、运行机制与管理建议。

补充说明：
- 本文档主要描述当前实现。
- 关于下一阶段的定价分层方案，见 [模型定价](./模型定价.md)。
- 关于下一阶段的结构治理，本文档第 9 节给出建议方案。

## 1. 分组是什么

分组不是展示字段，而是系统的核心策略维度，用于：
- 控制用户可用模型范围
- 控制请求可路由到哪些渠道
- 控制计费倍率（组倍率）

一句话：`分组 = 权限边界 + 路由边界 + 计费边界`。

## 2. 分组在系统中的作用

### 2.1 路由控制
- 用户发起模型请求时，系统先读取用户所属分组。
- 再按 `group + model` 从能力表中选择可用渠道。
- 若该分组下没有可用渠道，会直接返回“当前分组下无可用渠道”。

### 2.2 模型可见性控制
- 用户在前端可见的模型列表不是全量，而是按用户分组过滤后的结果。
- 同一套系统中，不同分组可见模型可以不同。

### 2.3 计费倍率控制
- 实际扣费 = `基础价格 × 分组业务倍率`。
- 基础价格来自供应商默认价或渠道覆盖价。
- 分组倍率来自 `groups.billing_ratio`，可用于实现会员、企业、内部等差异化计费策略。

## 3. 核心数据结构

### 3.1 用户分组
- 表：`users`
- 字段：`group`
- 含义：用户所属分组（如 `enterprise`、`internal`）

### 3.2 分组目录
- 表：`groups`
- 主要字段：`id`、`name`、`description`、`billing_ratio`、`enabled`、`sort_order`、`source`
- 用途：管理端展示与维护分组元信息

### 3.3 分组与渠道绑定
- 接口：`GET/PUT /api/v1/admin/group/{id}/channels`
- 含义：管理员在分组管理页显式选择该分组可使用哪些渠道
- 表：`group_channel_bindings`
- 当前落库方式：分组-渠道主关系先写入 `group_channel_bindings`
- 结果：系统会按 `group_models × group_channel_bindings × channel_models` 交集生成运行时能力映射

### 3.4 分组模型主表
- 表：`group_models`
- 关键维度：`group + model`
- 含义：某分组对外声明提供哪些模型，以及该模型归属哪个供应商
- 运行职责：
  - 决定 `/v1/models` 返回哪些模型
  - 为 `/v1/models.owned_by` 提供 `provider`
  - 作为运行时能力展开的主数据输入

### 3.5 能力映射（路由关键）
- 表：`group_model_routes`
- 关键维度：`group + model + channel_id`
- 含义：某分组是否可以通过某渠道使用某模型
- 运行职责：
  - 决定当前分组下某模型有哪些候选渠道
  - 记录该模型走该渠道时的 `upstream_model`
  - 记录分组内该渠道的优先级和启用状态
- 当前事实：这张表是运行时展开表，不再承担分组模型主数据语义
- 来源：由 `group_models`、`group_channel_bindings` 与 `channel_models` 交集批量刷新

### 3.6 模型供应商归属
- 表：`group_models`
- 字段：`provider`
- 含义：某分组下某模型的供应商归属
- 来源：由分组模型配置写入，迁移时从既有能力表回填
- 作用：
  - 为 `/v1/models.owned_by` 提供供应商归属
  - 显式校验“同一分组内同一模型只能归属一个供应商”

### 3.7 分组倍率
- 字段：`groups.billing_ratio`
- 用途：定义每个分组的业务倍率（默认通常为 1）

## 4. 请求处理流程（简化）

1. 鉴权后拿到用户 ID  
2. 根据用户 ID 查询 `users.group`  
3. 根据 `group + model` 在 `group_model_routes` 中选渠道（含优先级、可用状态）  
4. 发起上游请求  
5. 按 `基础价格 × 分组业务倍率` 计算消耗并记账

## 5. 管理接口（管理员）

当前分组管理接口位于：
- `GET /api/v1/admin/groups`：分页返回分组目录（支持 `page`、`page_size`、`keyword`）
- `POST /api/v1/admin/group`：新增分组
- `PUT /api/v1/admin/group`：更新分组（含启用状态）
- `DELETE /api/v1/admin/group/:id`：删除分组
- `GET /api/v1/admin/group/:id/channels`：读取该分组绑定的渠道
- `PUT /api/v1/admin/group/:id/channels`：更新该分组绑定的渠道

前端入口：管理员 Header 中“分组”Tab。

## 6. 约束与注意事项

- 正在被用户、渠道或能力映射使用的分组不允许删除。
- 仅创建分组目录并不会自动赋权；要让分组可用，需确保：
  - 用户被分配到该分组
  - 分组已绑定到合适的渠道
  - 能力映射已正确刷新
- 若修改了 `billing_ratio`，后续请求会按新的业务倍率计费。

## 7. 什么是运行时表

这里的“运行时表”是一个非常具体的概念：

- 指 Router 在真实请求进入后，为了完成选路而直接读取的表
- 服务的是在线请求处理阶段，不是管理端配置阶段
- 允许包含冗余字段，因为目标是降低请求时的查询复杂度

在当前实现里：

- `group_models` 是主数据表，表达“分组提供哪些模型”
- `group_channel_bindings` 是主数据表，表达“分组接入哪些渠道”
- `group_model_routes` 是运行时展开表，表达“这个分组下这个模型当前可走哪些渠道”

因此：

- `group_model_routes` 可以删除后重建
- 不允许管理端把 `group_model_routes` 当成主配置直接编辑
- 任何对分组模型或分组渠道的修改，都应先改主表，再刷新 `group_model_routes`

### 7.1 主值与运行时副本

当前这套结构里，有两类字段会同时出现在“主表”和“运行时展开表”中：

1. `priority`
2. `provider`

它们的规则是一样的：

- 主表里的值是配置真相
- `group_model_routes` 里的值是运行时副本

具体来说：

- `group_channel_bindings.priority` 是主值
- `group_model_routes.priority` 是运行时副本
- `group_models.provider` 是主值
- `group_model_routes.provider` 是运行时副本

这样设计的目的不是让两边各自独立维护，而是：

- 管理端只改主表
- 请求链路直接读 `group_model_routes`
- 重建链路负责把主值复制到运行时副本

因此如果两边不一致，应视为重建未完成或数据异常，而不是正常状态。

## 8. 当前实现的结构特点

### 8.1 `group_model_routes` 不是简单绑定表

当前代码里，`group_model_routes` 对应运行时主表 `GroupModelRoute`。它不是“分组和渠道的一条关系记录”，而是：

- `group + model + channel` 粒度的能力矩阵
- Router 实际选路时读取的候选池
- 分组内模型到上游模型的映射来源

这意味着：

- 只要删除一批 `group_model_routes` 行，该分组对应模型的路由能力就会一起消失
- “分组绑定渠道”这个产品动作，本质上会触发整组能力重建

### 8.2 `group_models` 已成为分组模型主表

当前实现中，不再单独维护 `group_model_providers` 表：

- `group_models.provider` 直接作为 `/v1/models.owned_by`
- `group_model_routes` 只保留运行时展开能力
- 同一分组内同一模型若出现多个 provider，仍视为配置错误

## 9. 推荐实践

- 保持分组数量适中，建议按业务等级或业务线分层：`free / pro / enterprise` 或 `internal / external`。
- 分组命名使用稳定英文标识（展示名称可中文）。
- 新分组上线前做三项检查：
  - 用户是否已分配
  - 渠道是否已绑定
  - 模型列表是否符合预期

## 10. 当前结构治理结论

### 10.1 `group_channel_bindings` 已落地

当前已经新增独立的分组-渠道关系表：

`group_channel_bindings`

建议字段：

- `group`
- `channel_id`
- `enabled`
- `priority`
- `created_at`
- `updated_at`

职责边界：

- 表达“这个分组接入了哪些渠道”
- 表达“这些渠道在该分组中的优先级”
- 不再让“分组绑定渠道”直接依附于 `group_model_routes` 的存在与否

### 10.2 三层职责拆分

治理后的职责建议如下：

1. `channel_models`
   - 表达渠道自身支持哪些模型、对应哪个上游模型、支持哪些端点
2. `group_channel_bindings`
   - 表达某分组接入哪些渠道，以及分组内渠道优先级
3. `group_models`
   - 表达某分组对外提供哪些模型，以及供应商归属
4. `group_model_routes`
   - 运行时展开表，只表达当前分组下模型实际可走哪些渠道

### 10.3 为什么要拆

当前没有独立 `group_channel_bindings`，会带来三个问题：

1. “分组绑定渠道”和“模型能力配置”混在一张表里，语义不清晰
2. 前端只要提交不完整渠道集合，就可能整体重建并误删能力
3. 无法干净表达“渠道已接入分组，但当前没有模型被启用”

### 10.4 对模型供应商归属的建议

当前建议：

- `provider` 以 `group_models.provider` 为主
- `group_model_routes.provider` 仅作为运行时冗余副本
- 不允许人工单独维护运行时副本

### 10.5 迁移顺序建议

本轮已经完成前两步，后续继续按下面顺序收口：

1. 已完成：新增 `group_channel_bindings` 表并回填现有分组-渠道关系
2. 已完成：分组渠道接口改为以 `group_channel_bindings` 为主关系
3. 已完成：新增 `group_models` 作为分组模型主表
4. 已完成：`group_model_routes` 改为由“分组模型 + 分组绑定渠道 + 渠道模型能力”生成
5. 已完成：`provider` 以 `group_models.provider` 为主
5. 已完成：删除 `group_model_providers` 独立表

### 10.6 建议表结构

```sql
CREATE TABLE group_channel_bindings (
  "group"      varchar(32)  NOT NULL,
  channel_id   varchar(64)  NOT NULL,
  enabled      boolean      NOT NULL DEFAULT true,
  priority     bigint       NOT NULL DEFAULT 0,
  created_at   bigint       NOT NULL DEFAULT 0,
  updated_at   bigint       NOT NULL DEFAULT 0,
  PRIMARY KEY ("group", channel_id)
);

CREATE INDEX idx_group_channel_bindings_channel_id
  ON group_channel_bindings (channel_id);

CREATE INDEX idx_group_channel_bindings_priority
  ON group_channel_bindings ("group", enabled, priority);
```

约束建议：

- `group_channel_bindings` 是分组接入渠道的主关系
- `group_models` 是分组模型主关系
- `group_model_routes` 只能为已存在于 `group_channel_bindings` 的渠道生成记录
- 关闭或删除某个分组渠道绑定时，应同步回收该分组该渠道下的模型能力

## 11. 常见问题

### Q1: 为什么用户看不到某个模型？
通常是该用户分组在 `group_model_routes` 中没有对应的 `group + model` 可用记录，或相关渠道被禁用。

### Q2: 为什么请求报“当前分组下无可用渠道”？
该分组下没有满足条件的启用渠道，常见原因是分组未绑定渠道、模型不匹配、或渠道状态禁用。

### Q3: 分组只是前端分类吗？
不是。分组直接影响后端路由决策、模型可见性与业务倍率计费。

### Q4: 模型供应商归属现在存在哪里？
当前主值存放在 `group_models.provider` 中；`group_model_routes.provider` 只是运行时冗余副本。

### Q5: 为什么还要新增 `group_channel_bindings`？
因为现在已经把“分组接入哪些渠道”和“分组提供哪些模型”拆成两张主表，`group_model_routes` 才能退回成纯运行时展开表。
